## 1. Summary

All student–AI exchanges are written to PostgreSQL by the AI proxy after each completed stream. This document covers the full schema, write path, consent enforcement query, and how conversation data is served to teachers and students.

---

## 2. Full Schema

```sql
-- Consent (one per student per project, permanent once created)
CREATE TABLE consents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    student_id      UUID NOT NULL REFERENCES users(id),
    project_id      UUID NOT NULL REFERENCES projects(id),
    agreed_at       TIMESTAMPTZ NOT NULL,
    agreement_text  TEXT NOT NULL,
    ip_address      INET,
    revoked_at      TIMESTAMPTZ,  -- retained for future policy; never set by app

    UNIQUE (student_id, project_id)
);
CREATE INDEX idx_consents_student_project ON consents (student_id, project_id);

-- One row per conversation session (conversation_id = client-generated UUID)
CREATE TABLE conversations (
    id              UUID PRIMARY KEY,
    student_id      UUID NOT NULL REFERENCES users(id),
    project_id      UUID NOT NULL REFERENCES projects(id),
    system_prompt   TEXT,         -- snapshot of AGENT.md at conversation start; admin-only
    started_at      TIMESTAMPTZ NOT NULL,
    last_message_at TIMESTAMPTZ NOT NULL,
    message_count   INTEGER NOT NULL DEFAULT 0,
    total_tokens    INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_conversations_student_project ON conversations (student_id, project_id);

-- One row per message (user prompt or assistant response)
CREATE TABLE conversation_messages (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL REFERENCES conversations(id),
    student_id      UUID NOT NULL REFERENCES users(id),
    project_id      UUID NOT NULL REFERENCES projects(id),
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant')),
    content         TEXT NOT NULL,
    model           TEXT NOT NULL,
    input_tokens    INTEGER,
    output_tokens   INTEGER,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_conv_msgs_student_project ON conversation_messages (student_id, project_id);
CREATE INDEX idx_conv_msgs_conversation    ON conversation_messages (conversation_id);
```

## 3. Write Path

The AI proxy writes logs as a FastAPI `BackgroundTask` after the stream completes:

```
Stream completes
  └─► background_tasks.add_task(write_log, ...)
        │
        ├─► UPSERT conversations (create or update last_message_at, message_count, total_tokens)
        │
        ├─► INSERT conversation_messages (user message)
        │
        └─► INSERT conversation_messages (assistant message)
```

**UPSERT pattern for conversations:**
```sql
INSERT INTO conversations (id, student_id, project_id, system_prompt, started_at, last_message_at, message_count, total_tokens)
VALUES (?, ?, ?, ?, now(), now(), 1, ?)
ON CONFLICT (id) DO UPDATE SET
    last_message_at = now(),
    message_count   = conversations.message_count + 2,
    total_tokens    = conversations.total_tokens + EXCLUDED.total_tokens;
```

This is append-only. No UPDATE or DELETE on `conversation_messages` is ever issued by the application.

## 4. Consent Enforcement

Every API endpoint that returns conversation data runs this check:

```sql
SELECT id FROM consents
WHERE student_id = :student_id
  AND project_id = :project_id
  AND revoked_at IS NULL
LIMIT 1;
```

- Result: row found → proceed
- Result: empty → return 403

This query hits `idx_consents_student_project` and returns in < 5ms. It runs in the AI proxy (before logging) and in the teacher/student API (before returning data).

## 5. Read APIs

### Teacher: get all conversations for a student project

```
GET /api/teacher/students/{student_id}/projects/{project_id}/conversations
  → consent check
  → SELECT conversations WHERE student_id=? AND project_id=? ORDER BY started_at
  → for each conversation: SELECT conversation_messages WHERE conversation_id=? ORDER BY created_at
  → system_prompt field omitted from teacher response (admin-only)
```

### Student: get own conversations for a project (IDE history panel)

```
GET /api/student/projects/{project_id}/conversations
  → JWT must match student_id (enforced by auth middleware)
  → same query, scoped to authenticated student only
  → system_prompt omitted
```

### Admin: full conversation detail including system prompt

```
GET /api/admin/students/{student_id}/projects/{project_id}/conversations
  → role=admin check
  → same query, system_prompt included
```

## 6. Conversation Boundary Semantics

The `conversation_id` is generated by the VS Code extension. When a student starts a new chat session, the extension generates a fresh UUID. The proxy trusts this value — it never merges conversations.

The teacher sees conversation boundaries in the timeline as labelled dividers based on gaps between `conversation.id` rows ordered by `started_at`.

---

## 7. Data Retention and Soft Delete

- `conversation_messages` rows are never deleted by the application
- If a student's account is soft-deleted (`users.deleted_at IS NOT NULL`), teacher API returns 403 for that student
- Admin API can still access logs for soft-deleted students
- No hard data deletion in v1

---

## 8. Trade-offs

| Decision | Alternative | Why chosen |
|---|---|---|
| Async background log write | Synchronous before stream | Non-blocking for student; acceptable best-effort loss on crash |
| PostgreSQL for logs | Dedicated log store (ClickHouse, OpenSearch) | Keeps stack simple; sufficient for school-scale data volumes |
| Append-only messages | Editable logs | Tamper-evidence; grading integrity |
| system_prompt in conversations table | In each message row | Single snapshot per conversation; reduces duplication |
